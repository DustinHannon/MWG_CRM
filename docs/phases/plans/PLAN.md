## 2. Locked tooling decisions (additional)

| Concern | Choice | Reason |
|---|---|---|
| Package manager | **pnpm** | brief §3 prefers pnpm; CLI present locally and Vercel auto-detects via `pnpm-lock.yaml`. Replacing npm `package-lock.json`. |
| Project config | **`vercel.json` only when needed** (cron stub Phase 7); no `vercel.ts` for v1 | Avoid bleeding-edge config surface for a freshly-scaffolded project. |
| Logging | structured stdout JSON; Better Stack adapter Phase 7 | Vercel captures stdout natively; gives us `get_runtime_logs` access. |
| Breakglass init | **`INSERT … WHERE NOT EXISTS … RETURNING id`** + unique partial index `users_one_breakglass` on `(is_breakglass) WHERE is_breakglass=true` | Idempotent, race-safe under concurrent cold starts. Generated password is logged exactly once because RETURNING only returns on the row that actually inserted. |
| DB credentials | Custom Postgres role `mwg_crm_app` (LOGIN, BYPASSRLS) created via MCP `execute_sql`; password set as a SCRAM-SHA-256 verifier (cleartext never logged). | Supabase Management API runs as a non-superuser `postgres` and cannot ALTER USER on privileged roles. Supavisor recognises any login role at the `<role>.<project_ref>` username format, so we don't need the actual `postgres` password. |
| Pooler hostname | `aws-1-us-east-1.pooler.supabase.com` (NOT `aws-0-`) | Newer Supabase projects are routed to the `aws-1-` Supavisor cluster. Connection format: `postgresql://mwg_crm_app.<ref>:<pw>@aws-1-us-east-1.pooler.supabase.com:<port>/postgres?sslmode=require`. Port 6543 = transaction mode, 5432 = session mode. |
| Direct connection | NOT used. POSTGRES_URL_NON_POOLING points at the **session pooler** (port 5432) instead. | `db.<ref>.supabase.co:5432` is IPv6-only by default; Vercel doesn't support IPv6 outbound without the IPv4 add-on. Session-mode pooler is IPv4 and behaves identically for `drizzle-kit generate`. |
| Drizzle driver | `drizzle-orm/postgres-js` with `prepare:false` (Supabase pooler) | Required to avoid prepared-statement collisions in transaction-mode pooler. |
| Migration runner | MCP `apply_migration` for tracked, server-managed migrations + `drizzle-kit generate` for SQL authoring | Brief mandate. |
| Auth | Auth.js v5 (`next-auth@beta`) + `@auth/drizzle-adapter`, `MicrosoftEntraID` + `Credentials` providers | Brief mandate. |
| UI primitives | `shadcn/ui` (Radix) on Tailwind v4, `lucide-react`, `sonner`, `next-themes` | Brief mandate. |
| Forms / tables | `react-hook-form` + zod resolvers, `@tanstack/react-table` | Brief mandate. |
| XLSX | `xlsx` (SheetJS) | Brief mandate. |
| Hashing | `@node-rs/argon2` | Brief mandate. |
| Date/time | `date-fns` + `date-fns-tz` | Brief mandate. |
| Fonts | Fraunces (display) / Geist (UI) / JetBrains Mono via `next/font/google` | Self-hosted to avoid runtime CDN fetches. |

## 5. Schema overview (Drizzle, `src/db/schema/*.ts`)

One file per logical area; barrel-exported from `src/db/schema/index.ts`. All UUID PKs default `gen_random_uuid()`; all timestamps `timestamp({withTimezone:true}).default(sql\`now()\`)`. Partial unique indexes for breakglass + Graph dedup.

Files:
- `enums.ts` — pgEnums for `lead_status`, `lead_rating`, `lead_source`, `activity_kind`, `activity_direction`, `import_status`.
- `users.ts` — `users`, `permissions`, plus Auth.js: `accounts`, `sessions`, `verification_tokens`.
- `leads.ts` — `leads` + indexes (owner, status, email, company, gin(tags), last_activity desc).
- `activities.ts` — `activities`, `attachments`.
- `imports.ts` — `import_jobs`.
- `audit.ts` — `audit_log`.

Schema details match brief §6 verbatim. No deviations.

## 9. Risk register

- **DB password capture via `ALTER USER`** — untested but standard Supabase pattern. If it fails I'll surface a single one-time SQL snippet for you to run in the Supabase SQL editor. Cost of failure: ~30s of your time.
- **Auth.js v5 still in beta** — published as `next-auth@beta`. The Drizzle adapter is `@auth/drizzle-adapter`. We'll pin exact versions to avoid surprise breaks during Phase 7.
- **Vercel function timeout for import** — default 300s on Fluid Compute; chunked 500-row batches stay well under that. If we hit it, I'll move large imports to a Vercel Queue (Phase 2 worth — out of v1 scope).
- **Graph permissions** — admin consent step in §16 is the gating risk; without consent the SSO flow returns tokens that can't read Mail/Calendars. Phase 3 will validate by hitting `/me` first, surface a clear error if Mail.Read fails.
